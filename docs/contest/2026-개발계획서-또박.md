# 2026 오픈소스 개발자대회 개발계획서 — 또박(Ttobak)

## 1. 프로젝트 개요

또박(Ttobak)은 어려운 한국어 공공·행정 문서(공문·고지서·안내문)를 발달장애인·경계선지능인·고령자·이주민·저문해자가 이해할 수 있는 "쉬운 정보(Easy-Read)"로 바꾸고, 그 **쉬움(easiness)**과 **사실충실성(fidelity)**을 기계가 재고, 그렇게 잰 위반(규칙 위반·충실성 위반)을 다시 생성 제약으로 되먹여 **스스로 고치는** 오픈소스 엔진이다. WCAG·점자·TTS가 정보의 "지각 가능성(perceivable)"을 다룬다면, 또박은 "이해 가능성(understandable)" 곧 인지 접근성을 겨냥한다.

## 2. 개발 배경 및 필요성

**문제 정의 — "읽을 수 있음"과 "이해할 수 있음"의 간극.** WCAG·점자·TTS가 다루는 것은 정보의 지각 가능성이다. 그러나 어려운 한국어 공공·행정 문서는 지각은 되어도 이해가 되지 않는다. 한자어, 긴 문장, 피동·부정·조건의 중첩, 전문 행정 용어가 인지 접근성 장벽을 만든다. 또박은 이 "이해 가능성" = 인지 접근성을 겨냥한다.

**왜 지금 필요한가.**

1. **법적 의무가 이미 존재한다.** 「발달장애인 권리보장 및 지원에 관한 법률」 제10조가 발달장애인이 이해하기 쉬운 형태의 정책정보 제공 의무를 부과한다.
2. **공급이 수요를 따라가지 못한다.** 공공정보 중 쉬운 정보로 제공되는 비율은 1% 미만으로 보고된다(웰페어뉴스·미디어생활, 2025-07-21). 반면 국가문해교육센터 2021년 조사에서 성인 약 20.1%가 낮은 문해력 구간에 속한다.
3. **격차를 메울 자동화가 없다.** 법은 의무를 부과했으나 실제 공급은 1% 미만이고 수요는 성인 5명 중 1명 규모다. 이 격차를 사람 손 제작에만 의존해 메우기는 현실적으로 어렵다.

즉 또박이 노리는 것은 "지침이 없어서"가 아니다. 이미 있는 지침을 자동으로, 수치로, 누구나 다시 돌려볼 수 있게 실제로 굴리는 수단이 없다는 것 — 그 빈자리를 채우는 데 있다.

## 3. 대상 사용자

또박은 IFLA의 Easy-to-read 두 집단 분류(영속적/일시적 인지 지원군)에 정렬해 대상을 설정한다.

**1차 수요자 — 최종 수혜자(읽는 사람).**

- 영속적 인지 지원 필요군: 발달·지적 장애인, 경계선지능인, 학습장애.
- 일시적·제한적 읽기 능력군: 고령자, 이주민·다문화, 저문해 성인, 신규 독자.
- 출력은 KEAD 2단계 모델에 따라 두 등급 지원 — **보통 읽기(Plain Language, 문해수준 3)** / **쉬운 글(Easy Korean, 문해수준 1–2, 레이아웃·여백·그림 중심)**.

**2차 수요자 — 도구를 직접 쓰는 생산측.**

| 사용자 | 사용 맥락 | 근거 |
|---|---|---|
| 공공기관 담당자(생산측) | 안내문·고지서를 쉬운 정보로 1차 변환 후 감수 | 법 제10조 의무 이행 주체 |
| 장애인·복지 단체, 도서관 | 쉬운 자료 제작 보조 | 현장 콘텐츠 제작 수요 |
| 개발자·에이전트 | LLM 워크플로우에 인지 접근성 변환을 임베드 | 재사용 가능한 오픈 코어/래퍼(MCP 서버 임베드는 권장 스트레치·미보장) |
| 연구자 | 공개 K-ER 지표·코퍼스로 한국어 Easy-Read 연구 | 오픈 데이터·측정 프레임워크 |

## 4. 개발 목표

### 4.1 정성 목표

- 쉬움과 충실성을 **동시에** 다루되, 충돌 시 항상 충실성이 이기는 Fidelity-first 설계를 관철한다.
- 품질을 "사람이 판정"이 아니라 "규칙 기반·재현 가능하게 측정"하는 층위로 전환한다.
- 코어·웹 데모·(스트레치) MCP 서버를 하나의 깨끗한 파이썬 코어 위에 얇은 래퍼로 얹어 재사용성을 보장한다.

### 4.2 정량 목표 — MVP 달성 현황과 8/27 목표 구분

| 지표 | MVP 달성 현황(현재) | 8/27 출품 목표 |
|---|---|---|
| 공개 함수(파이썬 코어) | 6개(parse·simplify·render_html·score·verify·match) 완성 | 유지·안정화 |
| 테스트 | 331개 통과 | 코퍼스 확장분 포함 회귀 유지 |
| 주석 코퍼스 페어 | **11쌍**(`synth-0001`~`synth-0011`, 전부 합성, 사람 검수 예정) | 유지·안정화 + 재배포 가능 KOGL-1 실문서 점진 보강 |
| 표면 | 코어 + 웹 데모(2표면) | 2표면 보장, MCP 서버는 권장 스트레치(3번째 표면) |
| 라이선스 감사 | `ttobak audit` 클린 | 유지 |

정직성 원칙상 밝힌다. **현재 코퍼스는 11쌍(전부 합성, 사람 최종 검수 예정)이며, 실측 K-ER은 원문 평균 72.0 → 쉬운 글 평균 81.8(Δ +9.8, n=11), 규칙 위반 평균 −2.45다.** 이 값은 손 주석이 아니라 `tooling/annotate_corpus.py`로 실제 엔진을 돌려 산출했고, 모든 gold 페어가 Fidelity 게이트 PASS(사실 보존)를 통과한다. 다만 (a) 문서가 합성이고 (b) 사람 라벨 대규모 벤치가 아니며 (c) 사람 최종 검수가 남아 있음을 함께 밝힌다. K-ER 점수는 규칙 기반 루브릭(비검증 보조 지표)이다. 실문서(KOGL-1)·100~300쌍 확장은 대회 이후 로드맵이며, 데이터셋 카드가 약속하는 n과 실제 평가 보고 n을 일치시킨다.

## 5. 주요 기능 및 특징

파이프라인은 **parse → simplify(생성→측정→교정 루프) → render**이며, 측정 단계에서 K-ER과 Fidelity 게이트를 호출한다. 6개 공개 함수는 모두 구현·테스트되었다.

1. **parse** — `parse(source, mime) → Document`. MIME 디스패치로 텍스트/PDF/HWPX를 공통 IR(Document/Block)로 변환한다. 구형 `.hwp` 바이너리는 거부하고, 저신뢰 추출은 confidence로 표기한다(graceful degradation).
2. **simplify** — `simplify(doc, level, provider, max_revise=3) → EasyReadResult`. LLM 생성 → K-ER 측정 → Fidelity 검증 → 위반·실패 슬롯을 하드 제약으로 되먹이는 자기교정 루프.
3. **render_html** — `render_html(result) → str`. 원문·쉬운본 나란히(side-by-side) 렌더, 상시 면책고지("쉬운본은 보조 자료이며 법적 효력은 원문이 우선합니다"), Fidelity 배지, 큰 글씨(18px) HTML.
4. **score (K-ER)** — `score(easy_text, level, source_text=None) → KERReport`. 규칙형 12규칙(R-01~R-12) 위반 체크리스트와 0–100 보조 점수. 관측 예시: 어려운 고지서 원문 score 56.5(level 3, 위반 8건) → 쉬운 재작성 score 83.3(level 1, 위반 5건).⁽¹⁾
5. **verify (Fidelity 게이트)** — `verify(source, easy_text, ref_date) → FidelityReport`. 숫자·금액·날짜·기한·자격·부정 슬롯 추출 → 정확일치 검증 → **PASS / REVISE / HUMAN_REVIEW** 라우팅. HIGH 슬롯 손상 시 PASS 불가.
6. **match (픽토그램)** — `match(easy_text) → list[PictogramRef]`. 약 30개 공공/행정 핵심개념 렉시콘 룩업, 글리프는 경로참조로만 반환(라이선스 인라인 임베드 금지). **현재 렉시콘은 Mulberry 매핑만 배선돼 있어 match()는 Mulberry 참조만 반환한다**(OpenMoji는 §9.2 참조).
7. **웹 데모** — `ttobak web`(Gradio). 업로드→parse→simplify→render 오케스트레이션 + 배지 표시, API 키 없을 때 FakeProvider 폴백.

> ⁽¹⁾ 56.5/83.3은 예시 문장쌍 1쌍을 1회 채점한 즉석 관측값(n=1)으로 코퍼스 등재값이 아니다. 코퍼스 차원 실측은 §9.4의 n=11 결과(평균 72.0→81.8, Δ +9.8)를 따른다. §10 머니샷의 79.3은 왜곡 주입 시나리오 관측값으로 또 다른 출처다.

## 6. 시스템 아키텍처 및 기술 스택

### 6.1 데이터 흐름

```
입력(bytes/str/Path, mime)
        │
        ▼
   parse() ──► Document(IR: Block[])            ← text/pdf/hwpx 디스패치
        │
        ▼
   simplify(doc, level, provider) ───────────────────────────┐
        ├─► provider.generate() → easy_text(초안)            │  교정 루프
        ├─► score(easy_text)    → KERReport(위반 R-01..12) ──┤ (max_revise=3)
        ├─► verify(doc, easy_text, ref_date) → FidelityReport ┤ 위반·실패슬롯을
        │      └─ router: PASS / REVISE / HUMAN_REVIEW        │ 하드 제약으로
        │   verdict==REVISE 이면 제약 주입 후 재생성 ─────────┘ 되먹여 revise()
        ├─► match(easy_text)    → list[PictogramRef]
        ▼
   EasyReadResult(source, easy_text, ker, fidelity, pictograms, revisions, verdict)
        │
        ▼
   render_html(result) → 원문+쉬운본 나란히 · 면책고지 · Fidelity 배지 HTML
```

웹 데모·평가 하네스는 위 6개 코어 함수의 얇은 래퍼다. `ttobak audit`는 릴리스 게이트로 코드트리 밖에서 병행한다.

### 6.2 모듈 계층 (실제 코드트리, 약 40개 `.py`)

| 계층 | 모듈 |

### 6.3 기술 스택

- **언어/런타임**: Python 3.11+, pydantic v2 전면, 테스트 pytest.
- **런타임 의존성**: pydantic, kiwipiepy(LGPL, 형태소 분석·지연 임포트), pypdf(PDF 주경로), pdfminer.six(PDF 폴백), hwp-hwpx-parser(Apache-2.0, HWPX), jinja2(렌더러), gradio(웹). PDF는 pypdf+pdfminer.six로 처리하며 PyMuPDF는 쓰지 않는다.
- **LLM provider(provider-agnostic, Protocol 추상화)**: 원격 Anthropic(가드형, 데모 기본), 로컬 Ollama(Kanana-1.5·Qwen2.5 등 Apache 사이즈, 지연 임포트), 테스트는 FakeProvider(결정론, 라이브 API 미사용).
- **미배선 후보(정직 구분)**: spaCy `ko_core_news_lg` NER, korean-number/es-hangul, FastAPI 등은 계획 후보이며 현재 코드에 미배선이다. Fidelity NER은 spaCy import가 가용할 때만 선택 적용되고, 결정론 regex/normalizer 코어가 권위 소스다.

## 7. 핵심 기술: 측정과 자가교정

이 절은 또박의 기술적 하이라이트다. 아래 동작·수치는 모두 저장소의 실제 코드/테스트를 실행해 확인한 값이다. 코퍼스는 현재 11쌍(전부 합성, 사람 검수 예정)이므로, 재현 가능한 실측이되 사람 라벨 대규모 벤치가 아님을 함께 명시한다.

### 7.1 K-ER — 규칙 기반 쉬움 측정

12개 규칙 함수가 각각 `RuleResult(sub_score 0~100, violations[])`를 반환하고, 그 단순 평균이 최종 점수가 된다(≥80→level 1, ≥60→level 2, else level 3). `metric/__init__.py`의 docstring이 명시하듯 이는 "규칙 기반 루브릭이며 경험적 검증이 아니다." 따라서 **하중 지지 산출물은 규칙별 pass/fail 체크리스트이고, 0–100 점수는 비검증 보조 지표**다. (§5.4의 56.5→83.3 예시는 위 각주 (1)대로 n=1 즉석 채점값이다.)

| ID | 측정 대상 | 임계값(코드) |
|---|---|---|
| R-01 | 문장당 형태소 수 | ≤15→100, ≥40→0, 선형 |
| R-02 | 쉬운 어휘 사전 밖 명사 비율 | 50%면 0점 |
| R-03 | 한자어·외래어 비율 | `hard_terms.txt` 멤버십 기반 |
| R-05 | 피동 비율 | ≤10%→100, ≥50%→0 |
| R-06 | 부정소 밀도 | 문장당 ≤0.5→100, ≥2→0 |
| R-07 | 괄호 설명 없는 전문용어 | 용어 뒤 `(`가 와야 면제, severity HIGH |
| R-11 | 직접 호칭(당신/여러분) | 존재를 가점 |

(전체 R-01~R-12 구현은 `metric/rules.py`의 `ALL_RULES` 레지스트리에 있다.)

### 7.2 Fidelity 게이트 — "슬롯 생존"으로 왜곡 검출

보존 필수 fact 슬롯을 규칙∪정규화로 추출한 뒤, 정량 슬롯(MONEY·DATE·NUMERIC·CONTACT·SCOPE 등 HIGH)이 쉬운본에 **정규화값 기준 verbatim 생존**했는지 검사한다. 핵심은 문자열이 아닌 값 비교라는 점이다. 원문 `30,000원`은 쉬운본의 `삼만원`으로도 생존한다.

라우팅 규칙(`router.py`)의 설계 요지는 **값 소실(복구 가능)은 REVISE, 의미 반전(값은 보존되나 뜻이 뒤집힌 왜곡)은 HUMAN_REVIEW**다.

```
if negation_flips or nli_contradictions or extraction_low_confidence: → HUMAN_REVIEW
if not failed_slots: → PASS
if any(실패 슬롯 not in _AUTO_RECOVERABLE): → HUMAN_REVIEW
else: → REVISE
```

경계어(이상=≥, 초과=>, 이하=≤, 미만=<, 까지=inclusive)는 값의 일부로 검사한다. 숫자는 그대로 두고 연산자만 뒤집는 왜곡은 정확일치로 안 잡히므로, 별도 `detect_drift_flips`가 피연산자(예 `65세`) 뒤 윈도우의 연산자 키워드를 원본 심볼과 비교해 잡는다. 단일 토큰 부정(제외/아니/불가/없/금지)은 NLI가 놓치기 쉬워 전용 NegationGuard가 규칙으로 잡아 HUMAN_REVIEW로 보낸다.

### 7.3 머니샷 재현 경로 — 어려운 고지서(`synth-0001`)

원장 소스: "건강보험료 1,295,400원을 2026년 7월 17일까지 강서구청에 납부…". 세 가지 왜곡이 게이트에서 각각 다른 경로로 처리된다(실제 `verify()` 테스트로 확인).

- **(a) 금액 반올림 '약 130만 원' → REVISE → 교정 → PASS.** 정규화 `1300000`이 원문 `1295400`과 불일치 → MONEY 슬롯 실패. MONEY는 자동복구 대상이므로 REVISE. 파이프라인이 "'1,295,400원'을 글자 그대로 포함, 반올림·의역 금지" 제약을 재주입해 재생성하면 PASS. (이 경로는 별도 테스트로 REVISE→PASS 라우팅만 확인하며, 확정 수치는 붙이지 않는다. 아래 결합 머니샷과 다른 시나리오다.)
- **(b) 자격 '이상↔이하' 플립 → HUMAN_REVIEW.** `만 65세 이상만` → `만 65세 이하만`. 숫자 65는 보존돼 정확일치는 통과하지만, drift-flip이 `>=` vs `<=`를 감지해 자동교정을 금지하고 사람 검수로 보낸다.
- **(c) 날짜 왜곡.** 코드가 실제로 잡는 것은 **날짜 시프트(7/17→7/7, REVISE)**와 **포함성 플립(까지→전에, HUMAN_REVIEW)**이다. "월만 남기고 일을 삭제"하면 원문 날짜가 사라져 REVISE로 라우팅되지만, 이는 "누락 복구" 경로이지 "모호화 의미 판정"은 아니다(전용 모호화 판정기는 없음 — 정직히 명시).

원장에 기록된 확정 수치 `revisions=1, K-ER=79.3, exact_fail=2`는 위 (a)와 (c)를 함께 넣은 **'약 130만 원 + 7월' 결합 머니샷**의 관측값이며(§10 런타임 검증 항목과 동일 시나리오), 반올림 단독 경로의 수치가 아니다.

`eval/`의 distortion-bench는 실제 `verify`를 7개 처리대상 왜곡 유형(number_swap, krw_unit_error, date_shift, inclusivity_flip, condition_flip, range_weaken, negation_drop)의 격리 코퍼스에 돌려 유형별 recall과 clean 대조군 FP=0을 검증한다. 단 이는 유형당 합성 1쌍 기준의 결정론적 단위 검증이며 사람 라벨 대규모 벤치가 아니다. `hallucinated_entity`(원문에 없는 기관명 추가)는 게이트 담당 범위 밖으로 명시 제외돼 있다.

## 9. 오픈소스 전략 및 라이선스

또박은 산출물 종류별로 라이선스를 분리해 오염(contamination)을 원천 차단한다.

### 9.1 산출물별 라이선스

| 산출물 | 라이선스 | 참조 방식 |
|---|---|---|
| 코드(코어·웹·MCP) | Apache-2.0 | 리포지토리 본문 |
| 데이터셋(쉬운 글·주석) | CC BY 4.0 | `corpus/` |
| 픽토그램 자산 | 각 세트의 CC BY-SA | `/assets/pictograms/` 아래 세트별 LICENSE와 함께, 경로/URL로만 참조 |

CC BY-SA 픽토그램은 Apache 코드나 CC BY 데이터에 인라인·base64로 절대 임베드하지 않는다. ShareAlike 조항이 코드·데이터로 전염되는 것을 막기 위한 조치다.

### 9.2 의존성 라이선스 매트릭스

픽토그램 세트는 **데이터/자산 계획**과 **코드 매칭 배선**을 구분해 표기한다. 현재 렉시콘(`pictogram/lexicon.py`)은 Mulberry 매핑만 배선돼 있고, `match()`가 반환하는 것도 Mulberry 참조뿐이다. OpenMoji는 코퍼스 seed(`synth-0001`)의 glyph 참조로만 존재하며 렉시콘(매칭 코드)에는 아직 미배선이다.

| 컴포넌트 | 선택 | 라이선스 | 배선 상태 | 회피한 대안(이유) |
|---|---|---|---|---|
| HWP/HWPX 파서 | hwp-hwpx-parser 1.0.0 | Apache-2.0 | 배선됨 | pyhwp = AGPL-3.0(네트워크 카피레프트가 Apache 배포와 충돌) |
| 형태소 분석 | kiwipiepy | LGPL-3.0 | 배선됨 | KoNLPy(+GPL 엔진) = GPL. LGPL은 별도·미변경·동적 임포트로만 사용 |
| 픽토그램(1차) | Mulberry Symbols | CC BY-SA 2.0 UK(상용 OK) | **렉시콘 매핑 배선됨(match 반환값)** | ARASAAC/KAAC = CC BY-NC-SA/NC(비상업이라 배포 불가) |

보조 의존성(pypdf·pdfminer.six·jinja2·gradio·dateparser 등)은 전부 permissive(BSD/MIT/Apache)로 clean 판정이다.

### 9.3 ttobak audit 게이트

`ttobak audit`(→ `tooling/check_licenses.py`)는 `pip-licenses` 파싱으로 다음을 fail-closed로 강제한다.

- **강한 GPL/AGPL/NC 차단, LGPL은 명시 통과.** `_is_strong_gpl`이 LGPL을 제외하고, 미분류 라이선스는 조용히 통과시키지 않고 "review needed(unrecognized-license)"로 표기한다. planted konlpy(GPLv3)·pyhwp(AGPL)·EXAONE(NC)를 차단하고 LGPL kiwipiepy는 통과시키는 것을 테스트로 단정한다.
- **프로젝트 클로저 스코핑.** `sys.executable -m piplicenses`로 감사를 돌리는 바로 그 프로젝트 venv만 조회한다. 공유 개발 머신의 무관한 AGPL/GPL 패키지는 클로저 밖이라 게이트에 걸리지 않는다.
- **자산 분리·시크릿/PII.** 코드/데이터 내 글리프 누출·base64 임베드·세트 LICENSE 누락, 그리고 AWS/Anthropic/OpenAI 키·개인키·한국 주민등록번호를 탐지한다.

현재 상태는 "PASS: license & security audit clean"(프로젝트 의존성 스코프)이다.

### 9.4 공개 코퍼스 계획

코퍼스는 CC BY 4.0으로 공개한다. 현재 **11쌍**(`synth-0001`~`synth-0011`, 전부 합성)의 완전 주석 source→easy-read 페어를 확보했으며(출품 목표 8~12쌍 충족), 각 페어는 K-ER + 필드별 Fidelity 주석 + 픽토그램 + 프로비넌스를 포함한다. 주석은 `tooling/annotate_corpus.py`로 실제 엔진을 돌려 도출하고 전 페어가 Fidelity PASS를 통과한다. 실측 코퍼스 K-ER은 평균 72.0→81.8(Δ +9.8, 위반 −2.45). 남은 일은 사람 최종 검수와 재배포 가능한 KOGL-1 실문서 보강이며, 100~300쌍 확장은 대회 이후 로드맵이다.

또한 코퍼스 제작 절차 자체가 게이트의 한계를 정직하게 드러냈다: 11쌍 전부 자동 Fidelity 게이트를 통과했지만, 별도의 **적대적 충실성 감사**가 그중 7쌍에서 게이트가 못 잡는 의미 결함(가능→명령 modality 뒤집힘, '소득인정액'→'소득' 개념 축소, AND 자격조건 분해, '검진기관'→'병원' 범위 축소 등)을 잡아냈고, 이를 개념 삭제 대신 풀어쓰기(글로싱)로 재작성해 해소했다. 자동 게이트(값 보존)와 사람/의미 검수(뜻 보존)가 상호 보완함을 보여준다.

**알려진 문서 이슈(정직한 원장).** 현재 `NOTICE`가 실제로 쓰지 않는 `soynlp`를 LGPL 의존성으로 과다 기재하고 있다(법적 위험은 아니나 원장 정확성 관점의 minor 불일치). 8/27까지 NOTICE 정리로 해소할 예정이다. 또한 라이선스 게이트가 `scripts/`와 `tooling/` 두 곳에 부분 중복돼 있어 통합 여지가 있다.

## 10. 개발 방법론 및 진행 현황

- **방법론**: SDD(spec/subagent-driven) + TDD. 각 태스크는 red→green→commit이며, 3-Phase 배치(Phase1 코어골격 T1–20 / Phase2 K-ER+Fidelity T21–35 / Phase3 표면·평가·라이선스 T36–63)로 impl→review→fix를 halt-on-failure로 진행했다.

이후 대회 일정과 정렬: 1차 서면평가(9/3) → 멘토링(9/18~10/9) → 출품작검증·기능테스트·라이선스검증(10/12~28) → 2차 발표평가(11/4) → 시상(12월 초). 특히 10월 라이선스 검증 구간은 `ttobak audit` 게이트가 이미 fail-closed로 대비돼 있다.

## 12. 기대 효과

1. **인지 접근성·정보격차 해소.** 법 제10조가 부과한 "이해하기 쉬운 정보" 의무를, 사람 손을 덜 타면서도 수치로 재고 다시 돌려볼 수 있는 방식으로 돕는다. 성인 약 20.1%의 낮은 문해력, 1% 미만의 쉬운 정보 공급이라는 격차를 좁히는 실무 도구를 지향한다.
2. **공무원·생산측 생산성.** 안내문·고지서를 쉬운 정보로 1차 변환한 뒤 감수하는 흐름을 지원해, 법적 의무를 지키는 초기 비용을 낮춘다. Fidelity 게이트가 왜곡을 자동으로 잡아내고, 확실치 않을 때는 사람 검수로 되돌리므로 안전하다.
3. **재사용성·연구 기반.** Apache-2.0 코어 + CC BY 4.0 코퍼스 + 공개 K-ER 측정 프레임워크로, 복지단체·도서관·개발자·연구자가 가져다 쓰고 넓힐 수 있는 공통 바탕을 제공한다. 특히 다시 돌려볼 수 있는 측정 프레임워크와 공개 코퍼스는, 한국어 Easy-Read 연구에서 아직 비어 있던 자리를 채운다.
